Pourquoi pas un seul grand tableau « à plat » avec tout dedans (comme on le ferait spontanément dans Excel) ? Parce qu'il devient vite énorme, redondant (la même info répétée sur chaque ligne), lent, et difficile à faire évoluer. Le modèle en étoile évite ces répétitions et donne de bien meilleures performances sur de gros volumes. La granularité (le niveau de détail d'une ligne de la table de faits) doit être choisie consciemment, dès la conception.
Why not a single big "flat" table with everything in it (as you'd naturally do in Excel)? Because it quickly becomes huge, redundant (the same info repeated on every row), slow, and hard to evolve. The star schema avoids this repetition and performs far better at scale. The grain (the level of detail of one fact-table row) should be chosen deliberately, from the start.
¿Por qué no usar una sola tabla «plana» con todo dentro (como se haría espontáneamente en Excel)? Porque rápidamente se vuelve enorme, redundante (la misma información repetida en cada fila), lenta y difícil de evolucionar. El modelo en estrella evita estas repeticiones y ofrece un rendimiento mucho mejor con grandes volúmenes. La granularidad (el nivel de detalle de una fila de la tabla de hechos) debe elegirse de forma consciente desde el diseño.

Checklist de robustesse

Robustness checklist

Lista de comprobación de robustez

  • Connecter directement les sources — pas de copier-coller
  • Toutes les transformations dans Power Query — pas dans Excel ni à la main
  • Modèle de données propre (relations, étoile) — pas un tableau à plat
  • Publier + planifier l'actualisation — pas de fichier envoyé par mail
  • Nommer clairement les étapes et les mesures — pour que ce soit maintenable
  • Connect sources directly — no copy-paste
  • All transformations in Power Query — not in Excel or by hand
  • Clean data model (relationships, star schema) — not one flat table
  • Publish + schedule refresh — no files emailed around
  • Name steps and measures clearly — so it stays maintainable
  • Conectar las fuentes directamente — sin copiar y pegar
  • Todas las transformaciones en Power Query — no en Excel ni a mano
  • Modelo de datos limpio (relaciones, estrella) — no una sola tabla plana
  • Publicar + programar la actualización — sin archivos enviados por correo
  • Nombrar claramente los pasos y las medidas — para que sea mantenible
À chaque réflexe correspond un piège classique : données copiées à la main (erreurs de saisie, oublis) ; transformations bricolées dans Excel (non reproductibles) ; un tableau unique géant (lent, redondant) ; un fichier renvoyé chaque semaine (versions multiples, perte de temps) ; des étapes nommées « Étape 1 », « Étape 2 »... (personne ne comprend plus le pipeline six mois après).
Each good habit has a classic trap on the other side: hand-copied data (typos, omissions); transformations hacked together in Excel (not reproducible); one giant flat table (slow, redundant); a file re-sent every week (multiple versions, wasted time); steps named "Step 1", "Step 2"... (nobody understands the pipeline six months later).
Cada buen hábito tiene su trampa clásica enfrente: datos copiados a mano (errores, olvidos); transformaciones improvisadas en Excel (no reproducibles); una sola tabla gigante y plana (lenta, redundante); un archivo reenviado cada semana (varias versiones, tiempo perdido); pasos llamados «Paso 1», «Paso 2»... (nadie entiende el proceso seis meses después).
